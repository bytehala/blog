WARNING: This post does not aim to inform in any way. However, I'd be happy if this inspires anyone (including me). This is part 2 of my narrative, if you care to read the first part, here it is.

Before you can run, you must learn how to walk.

I already told the story of how boring a task trying to learn TDD was for me. I mean this: WRITE THE TESTS FIRST BEFORE WRITING SOME CODE. Pfft! Come on, how ludicrous is that? I couldn't be bothered to do something so unnecessary.

Well, that's my excuse for not learning something that's actually useful because it's so hard.

Still, I hadn't given up on my Test Driven dreams. Or rather, it caught up with me.

So I strengthened my resolve: I was going to write tests no matter what.

I open Eclipse... *blink blink*... nothing! I don't know how to write a test. I don't even know how to use JUnit. I use my favorite ability: Google. Now I know how to do it. I start writing a test for... nothing. Now that is a really hard barrier. How can you test nothing? Later that day, in my search for "unit test exercises" I learn about kata from Peter Provost's blog. It's the only way to learn TDD, he claims. (I agree with him, btw.) He says you have to do kata for 30 minutes every day for the rest of your career, or 21 days at first. I also find out about Roy Osherove's unit testing book, and some mentions about a TDD lecture he did at NDC 2009. I promise to start doing kata the very same day. I did it 4 days later, the day after I watched Roy Osherove's lecture.

And I never looked back.
